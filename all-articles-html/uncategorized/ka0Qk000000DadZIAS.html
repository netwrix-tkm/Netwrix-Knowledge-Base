For FAST, your Netwrix Change Tracker must be version�7.0.1.51 or above.

The FAST service communicates with the Netwrix Cloud on port 8474

Trouble Shooting

If, after the successful installation of the FAST license and the creation of the FAST planned change, you find that new file integrity events remain in a processing and unknown state for an extended length of time, it is likely that Change Tracker is not successfully contacting the FAST Cloud.

If the Change Tracker installation is experiencing a high level of file integrity events then you may also find Change Tracker�s performance adversely effected. There should also be entries in the hubservice-log.txt indicating that FAST is unreachable.


If you have a customer�s installation exhibiting these symptoms, perform the following checks:

Recheck that the FAST license and the FAST planned change are correct using the details at the start of this document.

Confirm with the customer that FW rules to allow access to https://fast.nntsupport.com:8474 are in place.

Check that access is achievable by opening a browser on the Change Tracker server and pasting this link into the address bar.

Confirm with the customer if a web proxy server is in use. If a web proxy is used then it is likely that the above link test will be successful as the customer will be logged onto the Change Tracker server with an account which is allowed access through the web proxy. The status of TLS can be checked in the registry by reviewing the below registry path.

HKey_Local_Machine\System\CurrentControlSet\Control\SecurityProviders \SCHANNEL\Protocols

Typically, this key contains the following subkeys:

  • PCT 1.0
  • SSL 2.0
  • SSL 3.0
  • TLS 1.0

Each key holds information about the protocol for the key. Any one of these protocols can be disabled at the server. To do this, you create a new�DWORDvalue in the server subkey of the protocol. You set the�DWORDvalue to "00 00 00 00."

Currently, (5th April 2018), should TLS 1.0 be disabled then FAST will not work. The protocol can be reenabled by follwoing the details in the below MS article.�

https://support.microsoft.com/en-gb/help/187498/how-to-disable-pct-1-0-ssl-2-0-ssl-3-0-or-tls-1-0-in-internet-informat

Note: It is likely that TLS 1.0 has been diabled on purpose due to high profile vulnerabilitys, PCI expressly forbids the use of TLS 1.0 and so be sympathetic to understanding to the customer if you have to re-enable the protocol.
