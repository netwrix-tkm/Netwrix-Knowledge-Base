Summary: SIEM event come only from one of two identical AD policies.

Issue: Summary: �SIEM event come only from one of two identical AD policies.
Before fix:
�The reason of this problem is that Queue-based EventConsumers (like SIEM, SMTP, DotNetScript, and PowerShellScript) used the same object from the queue. Therefore one event consumer for one policy will put it's policy name and some other event consumer for different policy will get this changes which override correct value.
After fix:
Now every Queue-based EventConsumers will receive a clone of the object instead of object itself. So, all modifications made inside this clone will not influence other event consumers.
