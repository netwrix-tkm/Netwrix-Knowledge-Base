Migrate Threat Prevention Databases

Summary: How to Migrate Threat Prevention Databases

Issue: How to move your Threat Prevention databases to a new SQL server.  

Note: the agents will queue events until the enterprise manager is back online.  So, you want to minimize the down time to avoid queue overflows and missed events.  It may take a while for the enterprise manager to catch up on sending events to the database after the service is back online.  

Instructions:

  1. Stop the Enterprise manager, Reporting, and Scheduler services
  2. Use backup / restore or detach / attach methods for moving the databases to the new SQL server
  3. Ensure account(s) have appropriate rights on the new database
  4. Uninstall Threat Prevention
  5. Reinstall Threat Prevention, targeting the new SQL Server
  6. Start the services


Important note on permissions: 
If new permissions need to be applied, for some reason, make sure that all users connecting to the Threat Prevention database have:

  1. Database owner rights on the SI database
  2. a default schema of dbo
