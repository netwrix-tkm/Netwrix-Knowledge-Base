0-CreateSchema Long Runtime Post-Upgrade to NAA v12.0

IMPORTANT: This is a work-in-progress article. The product R&D team has yet to review some or all of the information below.

Symptom

After upgrading Netwrix Access Analyzer (NAA) from v11.6 to v12.0. the File System 0-CreateSchema job takes a long time to complete.



Cause


When upgrading from NAA 11.6 to 12.0, the File System 0-CreateSchema job is expected to take longer than it has in previous versions.  Analysis task "18. Data Migration" is moving IDs and metadata from the SA_FSAA_Resources table into separate, more organized tables, then removing the original columns. It also fixes table mappings for SA_FSAA_Trustees and SA_FSAA_Gates  by replacing missing values with default values if they haven't been filled yet. For large environments, or environments with large amounts of File System data, this can take 12+ hours.


Resolution


To address these causes, refer to the following steps.


  1. Schedule the File System 0-CreateSchema (no trigger needed).
  2. Disable all other File System jobs in the Scheduled node.
  3. From the Schedules node, right-click the 0-CreateSchema task and select Run.
  4. Once the 0-CreateSchema job has completed, re-enable all other File System jobs.
